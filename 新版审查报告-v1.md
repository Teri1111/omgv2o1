## 〇、一句话总评

**当前系统架构已从 T5 短路改为 LLM-First，但 LLM 集成仍然是"形式上的"——LLM 调用在 97.6% 的情况下回退到启发式，实际推理仍由规则驱动。测试流程仅有闭环 HIT 率统计，缺少论文级的消融实验、F1 指标、搜索策略对比和 Case Study。要达到可发表水平，需要从四个维度补齐：Prompt 工程、搜索机制、评估体系、训练/蒸馏流程。**

## 一、当前系统核心问题诊断

### 1.1 LLM 调用有效率极低（2.4% function call 成功率）

| 指标 | 值 | 严重度 |
|------|-----|--------|
| Function call JSON 解析成功率 | 2.4% (18/736) | **致命** |
| ReAct 格式回退覆盖率 | 97.6% | — |
| Empty LLM response 占比 | >50%（从 trace 分析） | **致命** |
| 模型直接回答问题而非调用工具 | 频繁出现 | **严重** |

**根因分析**：

1. **使用 Completion API 而非 Chat API**：deepseek-r1:14b 是 chat/instruct 模型，在裸 completion 模式下缺少 system/user/assistant 角色划分，指令遵循率极低。KBQA-o1 和 KnowCoder-A1 均使用 chat 格式。

2. **max_tokens=256 严重不足**：deepseek-r1 是推理模型，输出包含 `<think>...</think>` 内部推理，256 token 经常在 think block 内就被截断，导致有效输出为空。

3. **Scratchpad Token 膨胀**：Observation 中包含完整的 KB consultation JSON（含 embedding_text、steps 等冗余字段），3 步后 scratchpad 可达数万 token，导致后续 LLM 调用 prompt 超限。

4. **Prompt 格式与模型能力不匹配**：当前 prompt 混合了 function calling JSON 格式说明和 ReAct 文本格式，两种格式要求同时出现在同一个 prompt 中，模型无法明确该用哪种格式回复。

### 1.2 与 KBQA-o1 的 Thought/Action/Observation 格式对比

**KBQA-o1 的格式（用户提供的示例）**：
```
Thought1: At this step, we should identify a topic entity...
Action1: Extract_entity [ Taylor Lautner ]
Observation1: expression = START('m.07ldhs')
Thought2: At this step, we should find the one-hop relation...
Action2: Find_relation [ film.actor.film ]
Observation2: expression = JOIN('(R film.actor.film)', expression)
...
Action7: Finish [ expression ]
Observation7: expression = STOP(expression)
```

**omgv2-o1 当前实际输出**：
```
[empty response 或直接回答自然语言问题]
→ fallback to heuristic rule
```

**关键差距**：
- KBQA-o1 的 Thought/Action 由 **SFT 微调后的 Simulate Model** 生成，模型已经过数万条格式化轨迹的训练，格式遵循率接近 100%
- omgv2-o1 使用通用 LLM（deepseek-r1:14b 或 qwen2.5-coder:14b），未经 KBQA 领域微调，格式遵循完全依赖 prompt engineering
- KBQA-o1 的 Observation 是**简洁的表达式变化**（如 `expression = JOIN(...)`），而非 JSON blob

### 1.3 实际数据流与设计数据流的偏差

**设计目标（KBQA专家审查报告）**：
```
问题 → T5 子图 → LLM Agent → Tool 调用 → Skill 读取 → 执行反馈 → 最终答案
```

**实际运行（200 样本测试）**：
```
问题 → T5 子图 → LLM 调用（97.6% 失败）→ 启发式规则选关系 → 子图遍历 → 最终答案
```

LLM 在数据流中**事实上被旁路**，系统退化为「T5 子图 + 启发式贪心遍历」，与 T4 的 PG 基线本质上相同。67.4% 的 LF HIT 主要来自子图质量和启发式规则，而非 LLM 推理贡献。

## 三、67.4% HIT 率的真实含义

### 3.1 与论文结果的可比性

| 方法 | 数据集 | F1 | 训练数据量 | 模型参数 |
|------|--------|-----|-----------|---------|
| KBQA-o1 (MCTS) | CWQ | ~53-58% (推算) | CWQ train 全量 | 8B SFT |
| KnowCoder-A1 | CWQ | **59.3 F1** | 6.7k 样本 | 7B SFT+RL |
| Pangu (sota) | CWQ | ~54% | — | — |
| **omgv2-o1** | **CWQ 200 子集** | **67.4% HIT** ≠ F1 | **0** (zero-shot) | **14B 通用** |

**重要澄清**：
- omgv2-o1 的 67.4% **HIT 率**不等于 **F1 Score**。HIT = "golden 答案在预测答案集中"，这是一个宽松指标
- 当 HIT@1 仅 10.5% 时，说明大量 HIT 样本返回了多余答案，Precision 很低
- 如果按 F1 计算，实际分数会显著低于 67.4%
- 而且测试在 200 样本子集上进行，不是 CWQ test 全集（1639 样本）

### 3.2 性能归因问题

67.4% HIT 率的功劳主要来自：
1. **T5 子图质量**：已经将 6000+ relation 压缩到 3-15 条，搜索空间极小
2. **启发式规则**：贪心遍历子图中的关系，方向由 execute_partial 验证
3. **execute_partial 剪枝**：每步执行验证，自动排除无效关系

LLM 的实际贡献≈0（2.4% function call 成功率 × 部分有效比例）。

## 五、结论与建议

### 5.1 当前系统的真实定位

omgv2-o1 当前是一个**「T5 子图引导的启发式 KBQA 系统」**，虽然有 LLM 集成的架构但 LLM 未真正参与推理。67.4% HIT 率的功劳归属于 T5 子图质量 + 贪心遍历 + execute_partial 剪枝。

### 5.2 最小可行论文要求

要达到论文级别，**最少需要**：
1. **让 LLM 真正起作用**：通过 Phase 1 修复，使 LLM 决策成功率 >50%
2. **消融证明 LLM 贡献**：LLM-First > PG-only 且差距具有统计显著性
3. **F1 指标 + CWQ 全集**：可与论文结果直接对比
4. **一个创新的搜索/蒸馏方法**：如 "Subgraph-Constrained Agent" + Best-of-N

### 5.3 推荐行动优先级

1. **立即**：Phase 1 全部（1-2 天）
2. **本周**：Phase 2 的 F1 指标 + CWQ 全集 + 消融框架
3. **下周**：Phase 3 Best-of-N + Phase 2 的 Case Study
4. **后续**：Phase 4 训练流程
