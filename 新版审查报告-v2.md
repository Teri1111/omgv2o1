## 〇、一句话总评

**当前系统的主要矛盾已经不是“LLM 不会按格式输出”，而是“系统正在用错误的代理目标去放大 LLM 的错误选择”。P1 已经解决了接口和格式问题，P2 已经补上了 F1 评估，但 A4 仍然显著退化，说明问题已经从 prompt 层转移到搜索目标、动作约束、状态建模和训练范式层。继续做 P1 式 prompt 微调，收益会非常有限。**

## 一、哪些旧结论已经过时

与 v1 报告相比，当前仓库已经发生了两个实质变化：

1. **P1 已落地，不再是 Completion API 时代**
   - reasoning/llm_agent.py 已经切到 chat completions。
   - 统一了 Thought/Action/Observation 风格 prompt。
   - 增加了 think block stripping、scratchpad slimming、统一 action parser。

2. **P2 已落地，不再只有 HIT**
   - evaluate.py 已实现标准 F1/P/R。
   - test_closed_loop.py 已支持 full eval mode。
   - A4_analysis_report_v2.md 已经给出 CWQ test 全集 3390 样本上的 F1/Precision/Recall/EM。

因此，v1 报告中“先做 Chat API、先补 F1、P1 做完后大概率会改善”的判断已经不成立。**最新事实是：P1 和 P2 做完以后，LLM+Beam-3 仍然从 0.3194 F1 退化到 0.2323 F1。**

## 二、当前系统的真实能力与真实边界

### 2.1 已经具备的能力

当前 omgv2-o1 已经不是一个“没有 Agent 框架”的原型，而是一个具备以下组件的可运行系统：

1. **T5 路径压缩 + 子图构建**
   - reasoning/subgraph.py 与 skills/path_to_lf.py 已经形成 T5 候选路径到受限子图、再到 path-guided draft LF 的链路。

2. **Greedy / LLMGuided / Beam 三套控制器**
   - reasoning/agent.py 提供 GreedyAgent 和 LLMGuidedAgent。
   - reasoning/beam_agent.py 提供 BeamAgent。

3. **原子 LF 构造与执行验证**
   - skills/lf_construction.py 已覆盖 START、JOIN、AND、ARG、CMP、TC、COUNT、STOP。
   - skills/execution_feedback.py 已提供 execute_partial、execute_final、evaluate_candidate_relation 三层验证接口。

4. **trace 与闭环评估**
   - reasoning/agent.py 中的 TraceCollector 已经记录步骤、验证、工具统计、KB 统计。
   - test_closed_loop.py 已具备 HIT、F1、EM 的输出能力。

5. **轻量 observation tool 链**
   - explore_neighbors、verify_expression、consult_experience、inspect_path 已有 adapter 和预算控制。

### 2.2 当前边界在哪里

但这些能力仍然停留在“工程上可跑”，没有进入“搜索与学习目标正确”的阶段：

1. **LLM 只是插入控制流，不是定义控制流**
   - LLM 负责建议动作。
   - 环境负责兜底、模糊匹配、回退启发式。
   - 最终系统更像“启发式主导，LLM 试图插嘴”，而不是“LLM 在有约束的环境中学习决策”。

2. **Beam 只是多保留几个候选，不是有效搜索**
   - A0 与 A3 几乎同分，说明 beam 宽度不是当前瓶颈。

3. **多工具链是提示增强，不是环境状态转移**
   - observation tool 会写入 scratchpad、设置 last_failure、记统计。
   - 但它们没有形成像 KnowCoder-A1 那样明确的 observation-reward-done 状态机。

4. **trace 很丰富，但没有转换成训练资产**
   - 当前 trace 主要服务调试和报告。
   - 尚未形成 rejection sampling、SFT 语料、RL rollout 数据流。

## 三、A4 退化的根因不是“LLM 没输出对”，而是“代理目标与评估目标相反”

这是本次审查最关键的结论。

### 3.1 根因一：当前执行分数把“答案多”当成更好

在 skills/execution_feedback.py 中，execute_partial 的 execution_score 逻辑本质上是：

- 有结果：分数随 num_answers 增长，上限到 1.0
- 无结果：0.0
- 非 join 语法通过但未执行：0.5

这意味着：

1. 一个返回 120 个答案的宽关系，会比一个返回 3 个答案的精关系分数更高。
2. 这个信号会直接进入 reasoning/beam_agent.py 的排序逻辑。
3. Beam 分支排序又显式使用 execution_score 和 num_answers 作为主序。

**这和 F1/Precision 的目标是反向的。**

在 KBQA 里，大答案集通常意味着：

- 关系过泛
- 约束不够
- 走到了错误但可执行的路径

而当前系统却把这种情况当成“更值得保留”的候选。A4 中 709.1 的平均答案数，本质上就是这个代理目标的直接外显。

### 3.2 根因二：LLM 幻觉被“模糊接受”，而不是被“硬惩罚”

当前 reasoning/llm_agent.py 与 reasoning/agent.py 有大量 fuzzy fallback：

1. parse_action 在精确解析失败时，会扫描文本中出现的 relation 子串。
2. join、arg、time_filter、cmp 在 dispatch 时，也会做 relation 的模糊匹配。
3. 这会把“模型输出了一个不精确或错误的 relation”变成“系统替它猜一个差不多的 relation”。

这带来两个后果：

1. **错误动作没有被清晰地识别为 invalid action**。
2. **宽泛 relation 更容易因为名称相近而被匹配进来**。

KnowCoder-A1 的环境设计恰恰相反：

- 格式错是 invalid
- 工具不存在或参数错是 ineffective
- 都要记惩罚

当前 omgv2-o1 没有做到这一点，所以模型错误不会被强约束，而是被环境“温柔地接住”，最后进入错误分支。

### 3.3 根因三：Observation 工具没有形成真正的环境反馈闭环

当前 observation tool 的真实作用主要是：

1. 追加一条 observation 到 scratchpad
2. 更新计数器
3. 有时设置 last_failure
4. 有时提供文本提示

但缺失了 A1 式环境中最重要的三件事：

1. **显式 reward**
2. **显式 done / continue 判断**
3. **显式 action validity / effectiveness 标志**

所以现在的多工具链更像“提示增强器”，而不是“能通过交互修正策略的环境”。

### 3.4 根因四：Beam-3 几乎无增益，说明问题不在搜索宽度

A0 与 A3 的差别几乎为零，已经证明：

1. 当前子图空间内，单纯增加 beam width 并不能找到更好的解。
2. 真实瓶颈不是“候选不够多”，而是“排序信号错误”。

所以接下来如果继续从 Beam-3 改到 Beam-5、Beam-8，大概率仍然不会带来本质提升。

### 3.5 根因五：系统仍然是 zero-shot LLM，未进入论文对标的训练阶段

对标两篇代码后，差距已经非常清楚：

| 维度 | KBQA-o1 | KnowCoder-A1 | omgv2-o1 当前 |
|------|---------|-------------|----------------|
| 动作生成模型 | 专门训练的 simulate model | 冷启动 SFT 后继续 RL | 通用 deepseek-r1 zero-shot |
| 搜索目标 | fast reward + terminal reward | per-step reward + outcome reward | execute_partial 的答案数代理 |
| 状态建模 | WorldModel + SearchConfig + MCTS | ToolEnv + multi-turn rollout | scratchpad + heuristic fallback |
| 训练数据 | 轨迹生成再 SFT | rejection sampling + curriculum RL | 无训练 |

**根本差距不在“谁的 prompt 更会写”，而在“对方已经把策略变成模型能力，我们还在拿通用模型临场猜”。**

## 四、对标 KBQA-o1 后，应该借鉴什么，不该照搬什么

### 4.1 应该借鉴的部分

1. **显式状态机**
   - KBQA-o1 不是把所有东西都塞进 scratchpad，而是显式维护 step_idx、blocks_state、functions_state、expression_id。
   - 当前仓库应把 SearchState 从 GreedyAgent/BeamBranch 中抽离出来，而不是继续在 agent.py 里堆字段。

2. **fast reward 与 terminal reward 分离**
   - KBQA-o1 用 heuristic intuition 作为快速评分，再用 goal reward 做终局修正。
   - 当前仓库最大的问题就是把 partial exec answer count 当成单一目标。

3. **动作合法性规则前置**
   - KBQA-o1 会明确约束何时允许 Extract_entity、Find_relation、Merge、Order、Compare、Finish。
   - 当前 omgv2-o1 虽然有 allow_count、allow_finish 等布尔门控，但 relation 选择和 action 纠错仍然过于宽松。

4. **搜索树而不是排序列表**
   - 目前 BeamBranch 只是候选列表元素。
   - 更接近 KBQA-o1 的做法，是给 branch 加 visit count、Q 值、prior、fast reward，再做 UCT 选择。

5. **成功轨迹导出**
   - KBQA-o1 的一个关键价值是用搜索生成高质量轨迹再喂给后续模型。
   - omgv2-o1 当前 trace 已经足够多，完全可以转成 SFT 数据，而不是只做报告。

### 4.2 不应该照搬的部分

1. **不必照搬 KBQA-o1 的整套模拟接口**
   - 当前仓库已经有 T5 子图压缩与 LF 执行链，不需要把整个工程重写成 KBQA-o1 原样。

2. **不必先上完整 MCTS 再修目标函数**
   - 如果排序目标仍然奖励大答案集，MCTS 只会更稳定地搜到错误路径。

3. **不必回退到旧版 prompt 体系**
   - P1 已经把格式问题基本解决，再回到 JSON function calling 或 completion API 没意义。

## 五、对标 KnowCoder-A1 后，应该借鉴什么，不该照搬什么

### 5.1 应该借鉴的部分

1. **ToolEnv 语义**
   - step(action_text) 返回 observation、reward、done、info。
   - info 里还有 action_is_valid、action_is_effective。
   - 这正是 omgv2-o1 当前缺失的“多工具交互被量化”的部分。

2. **invalid 与 ineffective 区分**
   - 当前系统把大量错误 relation 当作 fuzzy hit 或 heuristic fallback。
   - A1 的设计告诉我们：必须先把 invalid 和 ineffective 分开记，再谈 RL。

3. **结果监督优先于过程监督**
   - A1 的强项不是 prompt，而是 outcome supervision。
   - 当前 omgv2-o1 最适合走的路，也是先把最终答案质量、答案膨胀惩罚、工具有效率纳入 reward。

4. **冷启动数据 + 课程式 RL**
   - 先 rejection sampling 得到可学轨迹。
   - 再从格式奖励、工具有效率奖励，逐步过渡到 F0.5/F1 奖励。

5. **工具响应长度控制与批量执行**
   - A1 明确限制 tool response 长度，避免 context 污染。
   - 这和当前 scratchpad slimming 的方向一致，但应该系统化到 ToolEnv 层，而不是只在 agent.py 中截断字符串。

### 5.2 不应该照搬的部分

1. **不建议现在就把全部动作切成 A1 式 JSON tool call 格式**
   - 当前仓库的文本 Thought/Action 格式已经能解析，真正缺的是 reward 和 environment semantics，不是外层语法。

2. **不建议先搭完整 RL 框架再修推理控制器**
   - 如果动作空间、状态定义、奖励函数都不稳，直接接 RL 只会把噪声训练进去。

## 六、当前项目的正确演化方向：O1 式搜索控制 + A1 式环境反馈 + 自身 T5 子图压缩

这不是简单的“选 o1 还是选 a1”，而应该是三者融合：

1. **保留 omgv2-o1 现有优势**
   - T5 路径压缩
   - 受限子图
   - 现成的 LF 执行验证

2. **引入 KBQA-o1 的搜索控制思想**
   - 显式 SearchState
   - 合法动作集
   - fast reward 与 terminal reward 分离
   - UCT/MCTS 或至少 UCT-lite

3. **引入 KnowCoder-A1 的环境与训练思想**
   - ToolEnv 风格 step
   - invalid / ineffective / effective 三态
   - rejection sampling 冷启动
   - curriculum RL

目标形态不应再是：

Question -> LLM 直接猜 relation -> partial exec -> 排序

而应变成：

Question -> T5 subgraph -> legal action candidates -> policy choose action id -> env step -> reward/info -> search / training

## 七、当前最关键的 8 个问题清单

1. **排序目标反了**
   - 大答案集被奖励，Precision/F1 被伤害。

2. **relation 解析太宽松**
   - 幻觉被软落地，不会被硬惩罚。

3. **observation tool 没有 reward 语义**
   - 只有文本提示，没有策略反馈。

4. **Beam 没有真正解决探索问题**
   - A0≈A3 已经说明这一点。

5. **LLM 贡献与 PG fallback 贡献混在一起**
   - 实验归因不干净。

6. **trace 没转成训练数据**
   - 工程资产未转化为模型资产。

7. **缺少 dev-based decision gate**
   - 目前直接跑全集，缺少快速止损机制。

8. **T12-T15 的本地功能修补没有触及 A4 宏观退化根因**
   - 这些修补能让功能更完整，但不能从 23.2 F1 拉回到可发表区间。

## 八、战略建议

### 8.1 明确不建议继续优先做的事

1. 不建议继续以 prompt wording 为主线投入大量时间。
2. 不建议优先扩大 beam width。
3. 不建议再增加更多 observation tool 名称，而不先建立 reward 语义。

### 8.2 建议立即转向的主线

1. **先修目标函数和动作约束**
   - 这是 zero-shot 还能不能站住的前提。

2. **再做搜索控制器重构**
   - 把 beam 从“按答案数排序”改成“按可解释代理信号探索”。

3. **尽快启动轨迹数据化与冷启动 SFT**
   - 当前系统已经到了“纯 prompt 工程收益递减”的阶段。

4. **RL 只在 SFT 站稳之后再上**
   - 否则会把当前噪声策略固化。

---

## 九、最终判断

**omgv2-o1 现在最值得做的不是“继续把 LLM 接得更深一点”，而是“把系统从启发式+LLM 插件，升级成有正确目标函数的 KBQA agent”。**

如果一句话概括 v2 结论，那就是：

**P1 证明了格式不是主矛盾，A4 证明了 zero-shot 通用模型在错误代理目标下会稳定劣化；下一步应当转向 O1 式搜索状态机和 A1 式结果监督，而不是继续做 prompt 微调。**
