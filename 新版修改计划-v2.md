# omgv2-o1 新版修改计划 v2

> 创建时间：2026-04-21
> 目标：基于 A4 结果与代码现状，重构为“目标函数正确、可训练、可搜索”的 KBQA Agent
> 对标策略：
> 1. 借鉴 KBQA-o1 的状态机、搜索控制、轨迹导出
> 2. 借鉴 KnowCoder-A1 的 ToolEnv、奖励设计、冷启动 SFT、课程式 RL
> 3. 保留 omgv2-o1 现有的 T5 子图压缩与 LF 执行链

---

## 〇、总原则

这版计划不再以“继续优化 prompt”为主线，而以三条硬原则为主线：

1. **先修正代理目标，再扩大搜索。**
2. **先做严格动作约束，再做训练。**
3. **先把 trace 变成数据资产，再谈论文级提升。**

---

## 一、目标架构

目标形态如下：

Question
-> T5 path / restricted subgraph
-> legal action candidate set
-> policy model choose action id
-> ToolEnv step(observation, reward, done, info)
-> search controller (Best-of-N / UCT-lite / MCTS)
-> terminal execution
-> trace export -> SFT / RL

其中三类模块的职责明确分开：

1. **State / Env 层**
   - 负责合法动作、执行反馈、reward、done。

2. **Policy / Search 层**
   - 负责选择 action、探索 branch、排序 trajectory。

3. **Training / Eval 层**
   - 负责轨迹导出、冷启动、RL、消融和 case study。

## 二、Phase 0：先把 zero-shot 推理从“负收益”修到“至少不拖后腿”

### P0-T1：重写 partial execution 的排序信号

**目的**：消除“答案越多越高分”的反目标问题。

**当前问题**：
- skills/execution_feedback.py 中的 execution_score 对 num_answers 单调递增。
- reasoning/beam_agent.py 又按 execution_score、num_answers 排序。
- 结果是宽关系被稳定保留，导致答案膨胀。

**修改方案**：
1. 将 execution_score 改成 rank_score，并与 trace 中的统计分数分离。
2. 对 join 步骤使用“精确优先”的代理分数：
   - invalid 或 exec error：强负分
   - 空结果：中负分
   - 小而非空的结果集：正分
   - 大结果集：对数惩罚
3. 不再把 raw num_answers 作为 tie-break 主序。

**建议公式**：

```text
rank_score = valid_bonus + path_prior + semantic_prior - size_penalty

size_penalty 随 log(1 + num_answers) 增大
```

**涉及文件**：
- omgv2-o1/skills/execution_feedback.py
- omgv2-o1/reasoning/beam_agent.py
- omgv2-o1/reasoning/agent.py

**验收标准**：
1. 200 样本 dev 上，LLM 模式平均答案数明显下降。
2. Precision 相比当前 A4 有显著提升。
3. 若 F1 仍低于 A0，则问题转入训练，而不是继续调 beam。

---

### P0-T2：把 relation 选择改成“严格候选选择”，停止 fuzzy 接受

**目的**：让错误动作被明确识别为 invalid 或 ineffective，而不是被系统猜测修正。

**当前问题**：
1. reasoning/llm_agent.py 的 parse_action 有 substring fallback。
2. reasoning/agent.py 的 _dispatch_action 对 arg/time/cmp 也会做 fuzzy relation 补救。

**修改方案**：
1. LLM 不再直接输出 relation name，而输出候选编号或严格枚举值。
2. prompt 中把可选 relation 列表编号化：

```text
Forward candidates:
[F1] film.actor.film
[F2] film.performance.film

Allowed reply:
Action4: Find_relation [ F2 ]
```

3. parse_action 只接受：
   - 合法 action type
   - 合法候选编号
   - 明确的 STOP / FAIL / NEED_OBSERVATION
4. 删除 relation substring fallback。

**借鉴来源**：
- KBQA-o1 的合法动作集约束
- KnowCoder-A1 的 invalid action 惩罚思路

**涉及文件**：
- omgv2-o1/reasoning/llm_agent.py
- omgv2-o1/reasoning/agent.py

**验收标准**：
1. invalid action rate 可被单独统计。
2. 不再出现“LLM 说了模糊 relation，系统替它补成宽关系”的情况。

---

### P0-T3：把 observation tool 从“写笔记”改成“改状态”

**目的**：让 explore/verify/consult/inspect 真正影响后续动作空间与分支评分。

**当前问题**：
1. explore_neighbors 主要往 scratchpad 里塞 relation 文本。
2. verify_expression 主要设置 last_failure。
3. inspect_path 主要给 recommendation。
4. consult_experience 主要写 guidance_text。

**修改方案**：
1. 为 agent 新增显式状态字段：
   - candidate_forward_ids
   - candidate_reverse_ids
   - last_observation
   - action_validity
   - action_effectiveness
2. 约定四类 observation 的硬效果：
   - explore_neighbors：刷新候选 relation 子集
   - verify_expression：给出 revise required 标志和 penalty
   - inspect_path：生成 alternative branch seed
   - consult_experience：只影响 prior，不直接改动作
3. scratchpad 仅保留简短自然语言摘要，真正的控制信息放状态里。

**借鉴来源**：
- KnowCoder-A1 的 ToolEnv
- KBQA-o1 的 state/action 分离

**涉及文件**：
- omgv2-o1/reasoning/agent.py
- omgv2-o1/skills/tools/adapters.py
- 可新增 omgv2-o1/reasoning/state.py

**验收标准**：
1. observation 后的动作候选集合可被 trace 记录。
2. verify 失败后可以触发明确 revise 分支，而不是只有文本提示。

---

### P0-T4：建立“动作有效率”指标

**目的**：从现在的 answer-only 指标，扩展到 agent-level 诊断指标。

**新增指标**：
1. action_valid_rate
2. action_effective_rate
3. avg_answer_count
4. explosion_rate
5. empty_result_rate
6. fallback_rate

**借鉴来源**：
- KnowCoder-A1 的 valid / effective 区分

**涉及文件**：
- omgv2-o1/reasoning/agent.py
- omgv2-o1/test_closed_loop.py
- 可新增 omgv2-o1/scripts/analyze_traces.py 的扩展逻辑

**验收标准**：
生成一张 inference diagnosis 表，能明确区分：
- 解析错
- 工具错
- relation 过泛
- 空结果
- PG fallback

---

### P0-T5：建立 200 样本 dev gate

**目的**：避免每次都直接跑 3390 样本全集。

**修改方案**：
1. 固定一个 200 样本 dev split。
2. 每次大的推理策略改动先跑 dev。
3. 只有满足 gate 才跑全集。

**Gate 标准**：
1. F1 不低于 A0。
2. Precision 高于当前 A4。
3. 平均答案数不超过 A0 的 1.2 倍。

**涉及文件**：
- omgv2-o1/test_closed_loop.py
- 可新增 omgv2-o1/scripts/build_dev_split.py

### P1-T2：引入 FastReward + TerminalReward 二级评分

**目的**：把“快速筛选”与“终局质量”分开。

**FastReward 组成建议**：
1. action validity
2. action effectiveness
3. relation specificity
4. path hint agreement
5. verification feedback

**TerminalReward 组成建议**：
1. train/dev 时用 answer F1
2. test 时仅用于分析，不回写真实 reward

**借鉴来源**：
- KBQA-o1 的 intuition + goal reward 组合
- KnowCoder-A1 的 outcome reward 思路

**涉及文件**：

### P1-T3：实现 UCT-lite / MCTSAgent

**目的**：把 beam 从“排序保留”升级为“探索-利用平衡”。

**修改方案**：
1. 在现有 BeamBranch 基础上增加：
   - visits
   - q_value
   - prior
2. 实现 UCT 或 PUCT 选择：

```text
UCT = Q + c * sqrt(log(N) / n)
```

3. expansion 时使用 candidate action set，而不是把所有 relation 全铺开。
4. simulate 可以先用 heuristic rollout，后续再替换成 learned policy。

**建议新增文件**：
- omgv2-o1/reasoning/mcts_agent.py

**备注**：
如果 P0 后 dev 表明 Best-of-N 已足够改善，可以先上 Best-of-N，再决定是否投入完整 MCTS。

### P1-T4：先做 Best-of-N 作为廉价强基线

**目的**：在完整 MCTS 之前，先验证“多次受约束采样 + 正确 rerank”能否带来正收益。

**修改方案**：
1. 同一问题采样 N 条 trajectory。
2. 用新的 terminal reranker 选择最好的一条。
3. reranker 输入：
   - answer count
   - verify result
   - path hint alignment
   - relation specificity

**借鉴来源**：
- KnowCoder-A1 的多 rollout 思想

**涉及文件**：
- omgv2-o1/test_closed_loop.py
- 可新增 omgv2-o1/reasoning/reranker.py

**数据来源**：

**建议新增文件**：

### P2-T2：做 rejection sampling 冷启动

**目的**：像 KnowCoder-A1 一样，先把“会基本合法调用工具”的策略训出来。

**方案**：
1. 在 CWQ train 上用强模型或当前强基线生成多条轨迹。
2. 保留：
   - 最终答案正确
   - action_valid_rate 高
   - action_effective_rate 高
   - answer explosion 低
3. 形成 5k 到 10k 条冷启动样本。

**可选数据源**：
1. GPT-4 类模型
2. 更强的 R1 / Qwen 推理模型
3. PG/A3 成功轨迹自动转写

---

### P2-T3：训练一个专门的 action policy model

**目的**：不再让通用 deepseek-r1 zero-shot 决定 relation，而是让模型学会在受限动作集中选动作。

**建议**：
1. 优先训练中小模型作为 action policy，而不是继续把大推理模型硬塞进在线控制器。
2. 训练目标先聚焦：
   - 格式正确
   - 候选选择正确
   - 工具调用有效

**借鉴来源**：
- KBQA-o1 的 simulate model

**验收标准**：
1. cold-start policy 在 200 样本 dev 上超过 zero-shot strict policy。
2. invalid action rate 显著下降。

**前提 gate**：

## 八、关键决策门槛

### Gate A：P0 之后是否继续做搜索重构

如果 strict zero-shot policy 在 dev 上：

1. F1 仍比 A0 低 2pp 以上
2. 但 precision 和 avg_answer_count 已显著改善

则说明方向正确，可以进入搜索或训练。

如果 strict zero-shot policy 连 precision 都不改善，则暂停搜索改造，直接优先做冷启动数据和 SFT。

### Gate B：SFT 后是否投入 RL

如果 cold-start SFT 不能稳定超过 A0/A3，先补数据质量，不进入 RL。

如果 cold-start SFT 已经能带来正收益，再进入 A1 式课程式 RL。

## 九、这版计划与旧计划的核心差异

旧计划的主线是：

Prompt 重构 -> 评估补齐 -> 搜索升级 -> 训练

这条路径在 P1、P2 完成以后，被 A4 证明为不够。

新计划的主线改为：

**目标函数纠偏 -> 动作约束严格化 -> 环境语义建立 -> 搜索控制器重构 -> 冷启动训练 -> 课程式 RL**

也就是说，接下来最先要修的不是“说法”，而是“目标”和“控制”。

## 十、最终建议

如果只保留一句行动建议，那就是：

**先把系统从“奖励大答案集的 zero-shot LLM 插件”改成“惩罚无效动作和宽关系的受限策略环境”，然后再用这些环境反馈去做搜索和训练。**

这条路线最接近：

1. KBQA-o1 的搜索价值
2. KnowCoder-A1 的训练价值
3. omgv2-o1 自身已经存在的子图与执行链优势
