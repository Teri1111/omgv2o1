### 4.5 超时行为（重要）

**dd0a169 修改**: `llm_agent.py` 中 `_request_completion()` 对 LLM 超时和连接失败抛 `RuntimeError`，不再静默 fallback。

- 超时（>10s）→ `RuntimeError: LLM request timed out ...`
- 连接失败 → `RuntimeError: LLM connection failed ...`
- HTTP 状态码错误（4xx/5xx）→ 仍走 fallback（合理）

**效果**: LLM 不可用时测试直接报错崩溃，不会跑出无意义结果。按当前 T12-T15 验收口径，这种“超时直接退出”视为合理的测试终止策略，不再作为本轮阻断缺陷。

### 4.6 只关闭自己启动的服务/隧道

如果复用的是已经存在的 ollama 或 8002 隧道，不要关闭。只有你自己新启动的服务/隧道，测试结束后才关闭。

## 五、核心架构

### 5.1 当前数据流（LLM-First，已实现）

```
T5 路径对齐 → candidate_paths
    ↓
受限子图构建 → G_restricted
    ↓
Path-Guided draft（语法合法则保存为 hint，不短路）
    ↓
LLMGuidedAgent / GreedyAgent / BeamAgent step-by-step 推理
    ↓ 有结果 → 输出
    ↓ 无结果 → PG draft fallback → 输出
```

### 5.2 S-expression 与 SPARQL

```
function_list → functions_to_expression() → S-expression → lisp_to_sparql() → SPARQL → 答案
```

### 5.3 2026-04-20 真实运行诊断（必须先看）

当前真正的主问题已不是 T11 之前的 observation 通路，而是 **LLM 虽已接入但没有真正参与推理**。基于当前保留下来的新版审查结论，200 样本诊断口径如下：

- Function call JSON 解析成功率：`2.4% (18/736)`
- ReAct / 规则 fallback 覆盖率：`97.6%`
- LF HIT：`67.4%`，Merged HIT：`68.0%`
- HIT@1：`10.5%`
- 常见失败：`empty LLM response`、直接回答自然语言、输出格式与解析器不匹配

结论：当前系统在运行时更接近 **「T5 子图 + 启发式贪心遍历」**，而不是论文目标里的真正 LLM-first agent。67.4% HIT 主要来自子图质量、execute_partial 剪枝和启发式规则，不能直接视为 LLM 推理贡献，也不能等价替代 F1。

当前根因定位如下：

1. `reasoning/llm_agent.py` 仍使用 `/v1/completions` 裸 prompt，而 deepseek-r1 更适合 chat/instruct 形式。
2. `max_tokens=256` 对推理模型过低，`<think>` 或长思维输出经常截断有效 action。
3. scratchpad / observation 过长，尤其是 KB consultation JSON 注入导致 prompt 膨胀。
4. 当前混合 JSON function-call 与 ReAct 两种输出协议，模型难以稳定遵循。

旧的 T11 前统计仍可作为历史背景保留在相关审查报告中，但**不再是当前决策依据**。

## 九、多代理协作 SOP

| 角色 | 职责 | 权限 |
|------|------|------|
| **PM（总控）** | 定数据流、定接口、定验收标准 | 不写代码 |
| **Dev（开发）** | 把设计转为可运行代码 | 唯一改代码的角色 |
| **Test（测试）** | 跑 CWQ 测试、输出失败分类 | 只读 |
| **Review（审查）** | 看 diff + 失败根因 | 只读 |

迭代流程：PM 下发任务 → Dev 完成 → Test 验证 → Review 审查 → PM 综合决策

**原则**：最小闭环优先、拒绝开发自测、每次只修一个主要错误、并行只用于读操作。
